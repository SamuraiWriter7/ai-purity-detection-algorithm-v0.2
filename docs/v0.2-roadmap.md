# v0.2 Roadmap

## Status

Roadmap draft.

This document defines the proposed direction for **Purity Detection Algorithm v0.2**.

Version `v0.1.1` established the minimum documented, example-backed, schema-validated, and CI-tested structure for purity assessment.

Version `v0.2` should strengthen the algorithmic layer by improving:

- signal weighting
- confidence handling
- warning-flag interpretation
- recursive synthetic risk detection
- review routing
- downstream readiness
- relationship to CollapseMonitor
- relationship to Royalty / Allocation Readiness

This roadmap is not an implementation specification yet.

It defines the next design targets.

---

Version `v0.1.1` provides the following foundation:


- `warning_flags`
- input signal scores

- GitHub Actions validation

This means `v0.1.1` is the first stable draft of the purity assessment layer.

---

## 2. Goal of v0.2

The goal of `v0.2` is to move from a minimal purity-assessment structure to a more explicit algorithmic framework.

In short:

```text
v0.1.1 = minimum valid purity assessment object
v0.2   = weighted, review-aware, risk-sensitive purity assessment model
```

Version `v0.2` should answer the following questions:

1. How should each signal be weighted?
2. How should low-confidence assessments be handled?
3. How should recursive synthetic risk be detected?
4. When should review be required?
5. How should purity assessment connect to CollapseMonitor?
6. How should purity assessment support royalty-readiness review without directly determining payment?

---

## 3. Proposed v0.2 Pillars

### 3.1 Signal Weighting Model

The current scoring model uses a draft formula.

Version `v0.2` should define a clearer weighting model for:

- provenance evidence
- author declaration quality
- structural originality
- revision lineage
- citation transparency
- AI-pattern risk
- recursive synthetic risk
- structure-fingerprint distinctiveness

Proposed new document:

```text
docs/scoring-weighting-model.md
```

This document should define:

- signal definitions
- default weights
- optional domain-specific weights
- confidence-adjusted scoring
- minimum required signals
- rationale for each weight

---

### 3.2 Confidence Handling

Version `v0.1.1` includes confidence values.

Version `v0.2` should define how confidence affects downstream interpretation.

Proposed confidence bands:

```text
0.85 – 1.00 : high confidence
0.70 – 0.84 : moderate confidence
0.50 – 0.69 : low confidence
0.00 – 0.49 : unreliable / review required
```

Recommended behavior:

```text
high confidence
↓
may proceed to downstream review

moderate confidence
↓
may proceed with review recommendation

low confidence
↓
review required

unreliable confidence
↓
blocked from royalty readiness until reviewed
```

This prevents purity scores from being treated as final truth.

---

### 3.3 Warning-Flag Severity Levels

Version `v0.1.1` defines warning flags.

Version `v0.2` should classify warning flags by severity.

Suggested severity levels:

```text
info
warning
review_required
blocking
```

Example mapping:

```text
missing_provenance              → review_required
high_ai_pattern_similarity      → warning
declaration_conflict            → review_required
low_confidence_score            → review_required
recursive_synthetic_risk        → blocking or review_required
origin_unclear                  → review_required
royalty_readiness_blocked       → blocking
```

Proposed new document:

```text
docs/warning-flag-severity-model.md
```

---

### 3.4 Recursive Synthetic Risk Detection

Recursive synthetic data is one of the main risks for model collapse.

Version `v0.2` should define a more explicit detection model for recursive synthetic risk.

Potential input signals:

- high similarity to known AI-generated patterns
- lack of primary-source provenance
- repeated summarization structure
- absence of firsthand observation
- shallow citation chains
- derivative phrasing across multiple sources
- low structural originality
- multiple AI-rewrite indicators

Proposed risk categories:

```text
none
low
moderate
high
critical
```

Example output:

```yaml
recursive_synthetic_risk:
  level: "moderate"
  confidence: 0.68
  basis:
    - "high_summary_pattern_similarity"
    - "weak_primary_source_provenance"
    - "limited_structural_originality"
```

Proposed new sample:

```text
examples/purity-assessment.recursive-synthetic-risk.sample.yaml
```

---

### 3.5 Low-Confidence Assessment Example

Version `v0.2` should include an example where the system cannot make a strong assessment.

This is important because uncertain cases are common.

Proposed new sample:

```text
examples/purity-assessment.low-confidence.sample.yaml
```

This sample should demonstrate:

- weak provenance
- ambiguous author declaration
- moderate AI-pattern risk
- low confidence
- review required
- royalty readiness blocked or deferred

This strengthens the protocol by showing that the system can say:

```text
I do not know enough.
Review is required.
```

That is a feature, not a failure.

---

### 3.6 Review Routing Model

Version `v0.2` should clarify when purity assessments move to human or multi-wing review.

Review routing should be triggered by:

- low confidence
- missing provenance
- recursive synthetic risk
- declaration conflict
- royalty-readiness request
- high downstream impact
- disputed origin claims

Proposed review modes:

```text
none
recommended
required
blocking_review
```

Potential reviewers:

- human reviewer
- domain expert
- automated system
- multi-wing review
- governance reviewer
- dispute registry process

Proposed new document:

```text
docs/review-routing-model.md
```

---

### 3.7 Relationship to CollapseMonitor

The Purity Detection Algorithm evaluates individual sources.

CollapseMonitor evaluates the ecosystem.

Version `v0.2` should define the bridge between these layers.

```text
Purity Assessment
↓
Aggregated Purity Metrics
↓
CollapseMonitor
↓
Civilization Health Signals
```

Possible aggregate metrics:

- average `origin_purity_score`
- natural-data ratio
- synthetic-data ratio
- hybrid-data ratio
- recursive synthetic risk rate
- low-confidence assessment rate
- review-required rate
- epicenter production trend

Proposed future repository:

```text
collapse-monitor-threshold-model-v0.1
```

The v0.2 roadmap should prepare the data outputs needed by that repository.

---

### 3.8 Relationship to Royalty Readiness

Purity assessment should support royalty-readiness review.

It should not directly determine allocation.

Version `v0.2` should preserve this principle:

```text
Purity score
≠
automatic payment decision
```

Instead, purity assessment should help determine whether an Epicenter is ready for further review.

Suggested royalty-readiness states:

```text
not_ready
review_ready
ready
blocked
disputed
```

A high purity score may strengthen review readiness.

A low or uncertain purity score should trigger review.

A blocking flag should prevent automatic movement toward allocation.

## 5. Proposed Schema Extensions

The current schema validates the minimal purity assessment object.

Version `v0.2` may extend the schema to include:

```yaml
recursive_synthetic_risk:
  level: "none | low | moderate | high | critical"
  confidence: 0.0
  basis: []

warning_flag_details:
  - flag: "recursive_synthetic_risk"
    severity: "review_required"
    explanation: "string"

review_routing:
  mode: "none | recommended | required | blocking_review"
  reviewer_type: "human | automated | multi_wing | governance"
  reason: "string"
```

The schema should remain strict enough for validation, but not so rigid that future protocol development becomes difficult.

## 7. Proposed v0.2 Scoring Direction

The current draft model is:

```text
origin_purity_score =
  0.25P
+ 0.15D
+ 0.25U
+ 0.15R
+ 0.10C
+ 0.10(1 - G)
```

Where:

```text
P = provenance evidence score
D = author declaration score
U = structural originality score
R = revision lineage score
C = citation transparency score
G = AI pattern risk score
```

Version `v0.2` may add:

```text
F = structure fingerprint distinctiveness
S = recursive synthetic risk score
Q = confidence quality score
```

Possible future direction:

```text
origin_purity_score =
  weighted_signal_score
  × confidence_adjustment
  × recursive_risk_adjustment
```

However, `v0.2` should remain understandable.

The goal is not mathematical complexity.

The goal is auditability.

## 8. Non-Goals for v0.2

Version `v0.2` should not attempt to:

- prove legal authorship
- decide copyright ownership
- automatically assign royalties
- ban AI-assisted creation
- define universal originality
- create a production payment system
- replace human or multi-wing review
- claim perfect AI-generation detection

The protocol should remain a review-support and ecosystem-health layer.

---

## 9. Design Philosophy for v0.2

The central principle remains:

```text
Purity is not a moral judgment.
Purity is an ecological signal.
```

Version `v0.2` should make this principle more operational.

The purpose is to protect the source ecology of AI civilization without rejecting AI-assisted creation.

A strong v0.2 should support the following balance:

```text
AI assistance is allowed.
Primary sources must remain traceable.
Synthetic recursion must be monitored.
Uncertainty must trigger review.
Value circulation must not bypass governance.
```
